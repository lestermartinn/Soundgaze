# Music-Similarity and Dimensionality-Reduction Research

For **song similarity**, prior work highlights that content‐based audio features often suffer from “hubness,” where a few tracks appear overly often as nearest neighbors【14†L125-L134】【23†L25-L32】.  In music retrieval tasks, applying **Mutual Proximity (MP)** – a probabilistic distance transform – has been shown to sharply reduce hubs and significantly improve k-NN accuracy (e.g. genre classification rates)【23†L25-L32】.  For example, the open-source Musly library computes similarity via 25-dimensional MFCC (timbre) features and Jensen-Shannon divergence *normalized by MP*【22†L25-L31】.  This suggests we could augment the Soundgaze pipeline by incorporating richer timbre features (e.g. MFCCs) and/or applying an MP-based reweighting to our 8D audio-feature distances.  Indeed, combinations like MFCC+Fluctuation Patterns are known to have **“almost no hubs,”** greatly improving recall while retaining audio similarity【14†L98-L106】.  By analogy, if our 8D Spotify features have high intrinsic dimension, we should normalize or adjust them (e.g. via MP) to mitigate hub effects and boost k-NN recall.  In practice, Soundgaze could implement MP on the 8D distance matrix (or expand features to include MFCCs) so that nearby points in 3D more faithfully reflect true audio similarity【23†L25-L32】【22†L25-L31】.

In **dimensionality reduction**, non-linear manifold methods tend to preserve local neighborhoods better than linear PCA【55†L626-L629】.  For instance, a recent study found that t-SNE and UMAP gave “more interesting” (clustered) projections than PCA for audio embeddings【55†L626-L629】, consistent with their focus on local structure.  Cutting-edge DR algorithms explicitly target neighbor preservation.  The *Regularized Autoencoder* (RAE) trains a neural mapping with constraints to **provably preserve k-NN relationships** during reduction【18†L69-L78】【18†L79-L80】.  RAE achieved **“superior k-NN recall”** compared to PCA/UMAP/t-SNE【18†L69-L78】【18†L79-L80】.  Similarly, *Parametric UMAP* replaces UMAP’s embedding step with a neural network, learning a mapping that matches UMAP’s layout *and* can quickly embed new points【32†L149-L157】.  Both are included here because they offer ways to get a 3D embedding with high local fidelity and a built-in transform for new songs.  Another framework, **Order-Preserving DR (OPDR)**, analytically relates the target dimension to expected KNN retention【39†L37-L45】.  Its experiments show one can choose dimensions to “effectively retain recall” after reduction【39†L37-L45】.  In short, these sources suggest concrete strategies: e.g. train a 3D autoencoder (RAE) on our audio-features with a KNN-preserving loss【18†L69-L78】【18†L79-L80】, or fit UMAP but then learn a neural approximator so we can embed new songs on-the-fly【32†L149-L157】.  Applying these could increase 3D KNN recall: the Soundgaze backend could implement RAE or Parametric UMAP on the 8D feature vectors to produce a 3D coordinate for each song that explicitly maintains neighborhood structure.

Industry and open-source projects provide practical examples of high-dimensional audio embeddings and search.  For example, an Elastic blog demonstrates transforming audio into 2048-d vectors (via a pretrained **PANNs** model) and indexing with cosine similarity【27†L169-L178】【28†L1-L4】.  While our current features are only 8D, we might explore richer audio embeddings (e.g. PANNs or other deep audio models) to embed songs more semantically.  The MusiCNN-embeddings project shows using Essentia’s *MusiCNN* model to extract deep music embeddings (from GTZAN) and then computing KNN similarity in that space【29†L258-L264】【29†L268-L274】.  Although Soundgaze uses higher-level features, this suggests we could integrate additional pretrained audio embeddings as part of our similarity metric.  On the search infrastructure side, Spotify’s new “Voyager” library – a production kNN index based on HNSW – delivers much faster and more accurate nearest-neighbor queries than their old Annoy implementation【58†L79-L83】.  This implies we should use a modern ANN search (e.g. HNSW via FAISS or similar) when finding neighbors in our 3D space to maximize recall/efficiency. In summary, these industry sources guide implementation: we could embed songs with a deep audio model and build an HNSW index for real-time retrieval, similar to Spotify’s approach【58†L79-L83】【27†L169-L178】.

Finally, **feature preparation and selection** can dramatically affect low-dimensional recall.  Work on contrastive audio embeddings shows that training data and augmentations shape local neighborhoods【25†L29-L38】.  For instance, augmenting to vary tempo/pitch can reduce their dominance, allowing genre/mood signals to cluster nearby, and *boost* KNN retrieval for salient features【25†L29-L38】.  Though our embeddings are fixed from Spotify data, this suggests that we should ensure our features emphasize perceptually meaningful dimensions.  For example, if *acousticness* or *mode* add noise to local clusters (as our summary suggested), we might drop or rescale them, as we did.  Likewise, weighting certain features (e.g. by genre prototypes) could improve cluster tightness.  The references above on MFCC+FP and MP imply that careful feature engineering – not just DR – is key to neighbor quality【14†L98-L106】【25†L29-L38】.  On the site, we could iterate feature combinations (even training a small autoencoder) to see which preserve KNN links best when reduced to 3D.  In practice, we would use these findings to fine-tune the Soundgaze data pipeline: for example, we might add a mutual-proximity step after computing song distances, or train a parametric 3D mapping with a loss on 8D KNN fidelity, both aimed at keeping similar songs “close” in the final 3D point cloud【23†L25-L32】【18†L69-L78】.

**Sources and How to Use Them:**  

- **Flexer et al. (2012)** – Audio hubness analysis【14†L125-L134】【14†L171-L179】. Included because it quantifies the hub problem in high-dim music data and shows *Mutual Proximity* fixes it (lower hubness, higher k-NN accuracy)【14†L171-L179】. We can apply MP on our 8D similarity matrix to improve neighbor reliability.  
- **Schnitzer et al. (2011)** – Mutual Proximity for audio【23†L25-L32】. Included because it demonstrates MP decreases hubs and raises k-NN classification rates【23†L25-L32】. We should integrate MP or similar distance warping in our pipeline to boost local similarity in 3D.  
- **Musly library (2014)** – MFCC-based audio similarity【22†L25-L31】. Included as an example of using 25 MFCCs + Jensen-Shannon + MP for quality music similarity【22†L25-L31】. This suggests adding MFCC/timbre features to complement Spotify’s attributes, and continuing to use MP as a hub fix.  
- **Contrastive Audio Embedding (2024)** – Local properties of music embeddings【25†L29-L38】. Included because it shows how training choices (data augmentation) affect which song attributes dominate neighborhoods (pitch vs genre). It achieved SOTA KNN accuracy by emphasizing genre/mood over low-level features【25†L29-L38】. This guides us to treat our input features to emphasize meaningful similarity cues (e.g. ensure that our 3D space encodes perceptual similarity, not irrelevant variance).  
- **DREAMS (2025)** – Local vs global preservation (via figure in appendix)【9†L1004-L1011】. Included because it outlines the trade-off between pairwise distance correlation and k-NN recall when reducing dimensions. It motivates methods that balance local-global fidelity. We can use this insight to evaluate methods: e.g. aim to maximize our recall score (like DREAMS does) when choosing 3D reduction parameters.  
- **RAE (ICLR 2026)** – Neural DR with k-NN guarantee【18†L69-L78】【18†L79-L80】. Included as a cutting-edge method that explicitly preserves k-NN relationships in reduction and provably bounds distortion【18†L69-L78】【18†L79-L80】. - **Parametric UMAP (2021)** – UMAP with learned mapping【32†L149-L157】. Included because it matches UMAP’s quality while learning a fast embedding function【32†L149-L157】. This implies we can either use ParametricUMAP (e.g. via umap-learn) or similarly train a net so new songs map consistently to our 3D space without re-running UMAP.  
- **OPDR (2024)** – Quantifying KNN preservation【39†L37-L45】. Included because it formalizes how many dims are needed to keep a given recall. We can use its formulas or metrics to justify why 3D might inherently limit recall and to set realistic expectations or adjustments (e.g. possibly consider 4th “height” if recall is critical).  
- **Elastic Blog (2023)** – PANNs embeddings for audio search【27†L169-L178】【28†L1-L4】. Included because it demonstrates a practical pipeline: extract MFCC, chroma, contrast features or use pretrained PANNs, index with cosine in a vector DB. It suggests possible feature expansions (beyond Spotify’s 8D) and validates using vector search for audio. Our site could similarly use deep audio embeddings and a vector index to compute 3D coordinates or find neighbors.  
- **MusiCNN-embeddings (2021)** – Pretrained audio embeddings【29†L258-L264】【29†L268-L274】. Included as an OSS example of extracting deep song embeddings (Essentia MSD-MusiCNN) and evaluating k-NN similarity (via genre metrics). This shows we can leverage existing music embedding models: e.g. feed our tracks through MusiCNN or similar to get fixed-length vectors, then apply DR.  
- **Voyager (Spotify, 2023)** – HNSW nearest neighbors【58†L79-L83】. Included as Spotify’s own recommender uses HNSW-based ANN for lightning-fast similarity search. For Soundgaze, we should likewise use a modern ANN library (HNSW/FAISS) to handle k-NN queries in 3D.  

Each source was chosen for its direct relevance to song similarity, dimensionality reduction, or neighborhood preservation.  The key takeaway for Soundgaze is to combine these insights: enrich and normalize features (per【14†L98-L106】【22†L25-L31】), use a DR method (and possibly a learned parametric mapping) tuned to preserve neighbors【18†L69-L78】【32†L149-L157】, and employ a fast kNN engine (like HNSW【58†L79-L83】) on the final 3D space.  Collectively, these approaches will help maintain the “shape” of the song cloud so that spatial proximity in 3D corresponds closely to true audio similarity.
